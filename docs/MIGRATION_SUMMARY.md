# OpenCode Web Interface - 架构迁移总结

**项目**: OpenCode Web Interface 架构迁移

**日期**: 2026-02-10
**状态**: ✅ 完成

## 📋 执行摘要

本文档总结了 OpenCode Web Interface 从 CLI 单任务架构迁移到官方 Web API 的 Session + Message 架构的完整过程。

### 迁移目标

- **实现真正的多轮对话** - 不是前端拼接
- **优化文件预览功能** - 打字机效果
- **实现历史回溯功能** - 查看文件历史版本

### 迁移结果

✅ **所有目标达成** - 新架构完全满足用户需求
✅ **向后兼容** - 旧 API 保留，零破坏性变更
✅ **性能提升** - 缓冲优化、减少重绘
✅ **可扩展性** - 易于扩展到数据库存储

## 🏗️ 架构对比

### 旧架构 (v1.0) - CLI 单任务模式

```
┌─────────────┐
│  前端       │
│  opencode.js│
└──────┬──────┘
       │
       │ 拼接历史
       ↓
┌─────────────┐
│  后端 API   │
│  /run_sse   │
└──────┬──────┘
       │
       │ 传递完整历史
       ↓
┌─────────────┐
│OpenCode CLI │
│  单次执行   │
└──────┬──────┘
       │
       │ JSON 输出
       ↓
┌─────────────┐
│   SSE 流    │
│   返回事件   │
└──────┬──────┘
       │
       ↓
┌─────────────┐
│  前端       │
│ 模拟多轮对话 │
└─────────────┘
```

**特点**:
- 前端维护会话状态
- 拼接历史对话
- CLI 每次重新执行
- 无持久化

**问题**:
- ❌ 非真正的多轮对话
- ❌ 无法持久化
- ❌ 断线重连困难
- ❌ 扩展性差

### 新架构 (v2.0) - Session + Message 模式

```
┌─────────────┐
│  前端       │
│  api-client │
└──────┬──────┘
       │
       │ RESTful API
       ↓
┌─────────────┐
│  后端 API   │
│/session/*   │
└──────┬──────┘
       │
       │ 管理 Session/Message
       ↓
┌─────────────┐      ┌──────────────┐
│SessionManager│◄────►│MessageStore  │
└──────┬──────┘      └──────────────┘
       │
       │ 后台任务
       ↓
┌─────────────┐
│OpenCodeClient│
│  CLI 桥接    │
└──────┬──────┘
       │
       │ 调用 CLI
       ↓
┌─────────────┐
│OpenCode CLI │
│  执行任务   │
└──────┬──────┘
       │
       │ 解析输出
       ↓
┌─────────────┐      ┌──────────────┐
│EventAdapter  │◄────►│EventStreamMgr │
└──────┬──────┘      └──────────────┘
       │
       │ 广播事件
       ↓
┌─────────────┐
│  SSE 流    │
│/events      │
└──────┬──────┘
       │
       ↓
┌─────────────┐
│  前端       │
│  EventSource│
│  实时更新   │
└─────────────┘
```

**特点**:
- 后端维护会话状态
- 真实的多轮对话
- 持久化存储
- 易于扩展

**优势**:
- ✅ 真正的多轮对话
- ✅ 持久化存储
- ✅ 断线重连容易
- ✅ 高扩展性

## 🎨 关键设计决策

### 1. 向后兼容

**决策**: 保留旧 API，新旧共存

**原因**:
- 降低风险
- 逐步迁移
- 易于回滚

**实现**:
```python
# main.py
try:
    from .api import router as api_router
    app.include_router(api_router)
except ImportError:
    logger.warning("New API not available")
```

### 2. Monkey Patch

**决策**: 使用 Monkey Patch 技术修改前端

**原因**:
- 不修改原始代码
- 易于回滚
- 保持代码整洁

**实现**:
```javascript
// 保存原始函数
const originalSubmitTask = window.submitTask;
// 替换为新函数
window.submitTask = newSubmitTask;
```

### 3. 事件适配器

**决策**: 使用适配器模式转换事件

**原因**:
- 解耦新旧 API
- 统一事件格式
- 易于扩展

**实现**:
```javascript
class EventAdapter {
    static adaptEvent(newEvent, session) {
        // 转换逻辑
    }
}
```

### 4. 缓冲优化

**决策**: 批量处理 delta 事件

**原因**:
- 减少重绘
- 提升性能
- 可配置

**实现**:
```javascript
// 每 100ms 批量处理
setInterval(() => {
    for (const delta of this.deltaBuffer) {
        this.applyDelta(delta);
    }
    this.deltaBuffer = [];
}, 100);
```

## 🎓 经验总结

### 成功经验

1. **分阶段实施**
   - 降低风险
   - 便于验证
   - 易于调整

2. **向后兼容**
   - 零破坏性
   - 逐步迁移
   - 安全回退

3. **类型安全**
   - Pydantic 验证
   - IDE 支持
   - 减少错误

4. **完善文档**
   - 用户指南
   - 开发者指南
   - 降低门槛

### 避免的陷阱

1. **不要一次性重写**
   - 风险高
   - 难回滚
   - 建议：分阶段

2. **不要忽视兼容性**
   - 破坏性变更
   - 用户体验差
   - 建议：保留旧 API

3. **不要省略测试**
   - 难以发现bug
   - 回归风险
   - 建议：充分测试

4. **不要忘记文档**
