# 历史数据无法显示Bug修复方案

## 问题诊断

### 根本原因
系统存在两套存储架构：

1. **history.db (SQLite)**: 持久化存储
   - `sessions` 表: 会话元数据
   - `messages` 表: 对话消息
   - `steps` 表: 工具调用事件（write, edit, bash等）
   - `file_snapshots` 表: 文件快照

2. **MessageStore (内存)**: 运行时存储
   - SessionManager使用
   - 浏览器刷新后数据丢失

### 问题流程
```
用户点击历史会话
  → 前端调用 /opencode/session/{id}/messages
  → API调用 session_manager.get_messages(session_id)
  → MessageStore从内存读取
  → 内存为空 → 返回空列表 ❌
```

### 数据验证
```bash
Messages记录数: 38
Steps记录数: 37   ← 工具调用事件有数据，但无法显示
Sessions记录数: 16
```

## 解决方案

### 方案选择
**方案1**: 修改`get_messages`，增加数据库回退逻辑 ✅ 推荐
- 优点：按需加载，性能好，兼容性强
- 缺点：需要处理内存和数据库的数据同步

### 修复步骤

#### 1. 修改 MessageStore 添加数据库恢复方法

在 `app/managers.py` 的 `MessageStore` 类中添加：

```python
async def restore_session_from_db(self, session_id: str) -> bool:
    """
    从数据库恢复会话到内存

    Args:
        session_id: 会话ID

    Returns:
        是否成功恢复
    """
    try:
        import sqlite3
        from .models import Message, MessageInfo, PartType, ContentType

        # 初始化内存结构
        if session_id not in self.messages:
            self.messages[session_id] = {}
        if session_id not in self.parts:
            self.parts[session_id] = {}
        if session_id not in self.message_order:
            self.message_order[session_id] = []

        # 从数据库加载消息
        conn = sqlite3.connect('history.db')
        conn.row_factory = sqlite3.Row
        cursor = conn.cursor()

        # 读取messages表
        cursor.execute("""
            SELECT message_id, session_id, role, created_at
            FROM messages
            WHERE session_id = ?
            ORDER BY created_at ASC
        """, (session_id,))

        rows = cursor.fetchall()
        if not rows:
            conn.close()
            return False

        for row in rows:
            message_id = row['message_id']
            role = row['role']
            created_at = row['created_at']

            # 创建Message对象
            message = Message(
                id=message_id,
                role=role,
                timestamp=int(datetime.strptime(created_at, '%Y-%m-%d %H:%M:%S').timestamp())
            )

            # 存储到内存
            self.messages[session_id][message_id] = message
            if message_id not in self.message_order[session_id]:
                self.message_order[session_id].append(message_id)

            # 读取消息部分
            cursor.execute("""
                SELECT part_id, part_type, content_type, content, start_time, end_time
                FROM message_parts
                WHERE message_id = ? AND session_id = ?
                ORDER BY start_time ASC
            """, (message_id, session_id))

            part_rows = cursor.fetchall()
            for part_row in part_rows:
                from .models import Part, PartTime

                part = Part(
                    id=part_row['part_id'],
                    type=PartType(part_row['part_type']),
                    content_type=ContentType(part_row['content_type']),
                    content=part_row['content'],
                    time=PartTime(
                        start=int(part_row['start_time']) if part_row['start_time'] else None,
                        end=int(part_row['end_time']) if part_row['end_time'] else None
                    )
                )
                self.parts[session_id][message_id][part.id] = part

        conn.close()
        logger.info(f"Restored {len(rows)} messages for session {session_id}")
        return True

    except Exception as e:
        logger.error(f"Error restoring session from DB: {e}")
        return False
```

#### 2. 修改 SessionManager.get_messages

在 `app/managers.py` 的 `SessionManager` 类中修改 `get_messages` 方法：

```python
async def get_messages(self, session_id: str) -> List[MessageWithParts]:
    """
    获取会话的所有消息（按时间顺序）

    Args:
        session_id: 会话ID

    Returns:
        消息列表（包含部分）
    """
    # 如果会话不在内存中，尝试从数据库恢复
    if session_id not in self.message_store.messages:
        logger.info(f"Session {session_id} not in memory, restoring from database...")
        restored = await self.message_store.restore_session_from_db(session_id)
        if not restored:
            logger.warning(f"Failed to restore session {session_id} from database")
            return []

    return await self.message_store.get_messages(session_id)
```

#### 3. 添加steps数据返回API

创建新API端点返回steps数据（工具调用事件）：

在 `app/api.py` 添加：

```python
@router.get("/session/{session_id}/timeline")
async def get_session_timeline(session_id: str):
    """
    获取会话的时间轴（工具调用事件）

    Args:
        session_id: 会话ID

    Returns:
        时间轴事件列表
    """
    try:
        from app.history_service import get_history_service

        history_service = get_history_service()
        timeline = await history_service.get_timeline(session_id)

        return {
            "session_id": session_id,
            "timeline": [
                {
                    "step_id": step["step_id"],
                    "action_type": step["action_type"],
                    "file_path": step["path"],
                    "timestamp": step["timestamp"]
                }
                for step in timeline
            ],
            "count": len(timeline)
        }
    except Exception as e:
        logger.error(f"Error getting timeline: {e}")
        raise HTTPException(status_code=500, detail=str(e))
```

#### 4. 前端加载timeline数据

在前端添加timeline加载逻辑：

```javascript
// 在 opencode-new-api-patch.js 中添加
async function loadSessionTimeline(sessionId) {
    try {
        const response = await fetch(`/opencode/session/${sessionId}/timeline`);
        if (!response.ok) throw new Error('Failed to load timeline');

        const data = await response.json();

        // 渲染timeline事件
        data.timeline.forEach(event => {
            processEvent({
                type: 'tool_use',
                tool_use: {
                    name: event.action_type,
                    input: event.file_path ? {file_path: event.file_path} : {}
                },
                timestamp: event.timestamp
            });
        });
    } catch (error) {
        console.error('Error loading timeline:', error);
    }
}
```

## 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据库schema不匹配 | 高 | 先验证schema兼容性 |
| 内存占用增加 | 中 | 限制单次恢复的消息数量 |
| 性能下降 | 低 | 添加缓存机制 |

## 总结

- **问题**: MessageStore内存存储，刷新后丢失，缺少数据库恢复逻辑
- **方案**: 添加`restore_session_from_db`方法，在get_messages时按需恢复
- **优先级**: 🔴 高优先级 - 影响核心用户体验
